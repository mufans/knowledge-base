    Hermes 个人方向与机会发现系统设计¶

    文档状态¶

    • 日期：2026-07-19
    • 状态：已批准，进入实施
    • 实施方式：用户复核后，生成一份端到端实施计划，并在一次连续执行中完成
    • 取代：2026-07-19-hermes-knowledge-lab-design.md 仅作为历史技术试点方案，不再作为实施目标

    1. 背景与用户约束¶

    用户 36 岁，具有 12 年移动端开发经验，正在关注 AI Agent、HarmonyOS、Android、端侧 AI 与开发效能。当前没有希望被强制绑定的长期目标，最需要的是发现方向和机会。

    现实约束：

    • 当前合同计划于 2027-05 到期；
    • 无固定收入可支撑约 3 个月；
    • 现有 OpenClaw 已运行多类定时采集与提炼任务；
    • 现有知识库对 raw/ 只读，对 wiki/ 可写，并可能同步到 GitHub Pages；
    • SmartInspector、Skill、Hermes、OpenClaw 和知识库均为工具或试验载体，不得被设定为个人发展主轴；
    • 需要 API Key 时必须暂停，由用户在 PTY 隐藏输入中亲自粘贴。

    2. 系统目标¶

    系统的第一使命是：

    持续发现值得关注的方向和机会，通过低成本实验获得外部证据，增加用户的职业选择权、收入安全性、技术适应力和可复用个人资产。

    系统不预设用户必须成为某类工程师、继续某个项目、制作 Skill 或走独立开发路线。这些只能作为可验证的方向假设。

    3. 最高优先级原则¶

    3.1 禁止信息茧房¶

    • 采集准入不使用当前目标、历史点击、已有项目或个人标签作为过滤条件。
    • 个人匹配度只影响后续分析深度和行动优先级，不影响是否被采集。
    • 每日输出至少包含 1 条非当前关注领域的意外发现。
    • 保留新出现且尚未分类的主题、跨领域弱信号、反对证据和持续升温的已忽略方向。
    • 每季度对方向假设执行一次清零式复核，避免历史偏好形成自我强化。

    探索期的呈现配额是 40% 现有优势相关、40% 全域热门技术、20% 跨领域与弱信号。这是呈现多样性要求，不是采集白名单。

    3.2 观察开放，采用保守¶

    • 系统要及时知道最新变化，但不自动将最新版本视为推荐版本。
    • 技术状态同时维护 known_latest 和 recommended_stable。
    • 新发现先进入 Frontier，通过官方正式发布、文档完整性、最小兼容性测试、无严重已知问题和可回滚性检查后，才能进入 Stable。
    • 知识的复核时间用 review_due_at 表示，超期表示需要复核，不表示自动作废。
    • 实际环境更新必须经过验证并保留回退路径。

    3.3 事实、推断与假设分离¶

    • Fact：官方当前文档、官方 Release、官方代码或原始论文可验证。
    • Inference：基于多个信号的综合判断，必须标明推断。
    • Hypothesis：尚待实验的方向或机会。

    任何社区讨论、二手报道或热门观点均不得直接晋升为 Fact。

    4. 方案比较与选择¶

    4.1 信息雷达型¶

    优点是覆盖广，但与现有 OpenClaw 重复，且容易继续生产摘要而非方向判断。

    4.2 机会发现与实验型（采用）¶

    将广泛信号转化为机会假设，再使用 1–2 周的低成本实验获得外部证据。它不预设项目、Skill 或职业标签，最符合用户当前缺少明确目标的状态。

    4.3 高度自治创业型¶

    在缺少已验证方向且现金缓冲只有 3 个月时，自动开发、发布与营销的风险和噪声过高，不采用。

    5. 系统边界与职责¶

    5.1 OpenClaw¶

    • 作为唯一生产定时调度中心；
    • 负责广域信号采集、原始来源保存、基础去重和重要变更事件发现；
    • 不做个人方向决策；
    • 不向 Hermes 迁移或复制生产 Cron、钉钉凭据和运行状态。

    5.2 Hermes¶

    • 使用独立私有 Profile 运行；
    • 负责趋势聚类、机会卡片、方向组合、实验设计、证据跟踪和周期复盘；
    • 负责技术知识的 Frontier/Stable 状态和使用前高影响复核；
    • 默认只读访问知识库；
    • 不接管生产采集，不自动对外发布、投递、联系他人、付费或删除。

    5.3 知识库¶

    • 继续作为技术事实、来源摘要和跨源综合分析的长期仓库；
    • 严格遵守 raw/ 只读、wiki/ 可写和日志规则；
    • 不保存现金缓冲、求职反馈、私人联系人、未公开机会和个人职业判断。

    5.4 用户¶

    • 保留方向选择和风险决策权；
    • 确认密钥、外部发布、外部沟通、付费、删除和重要代码操作。

    6. 核心数据对象¶

    Hermes 私有层使用以下逻辑对象：

    • Signal：单个外部信号，包含来源、日期和可靠性。
    • Trend：多个信号形成的持续变化。
    • Opportunity：可验证的职业、技术、产品、服务、开源、内容、人脉或跨领域机会假设。
    • Experiment：1–2 周内可完成的最小验证。
    • Direction：处于观察、验证或主动投入状态的方向。
    • Evidence：支持或反对某项判断的可追溯证据。
    • Decision：继续、调整、暂停或停止及当时依据。
    • Review：周、六周或季度复盘。
    • TechState：技术的 known_latest、recommended_stable、成熟度、复核时间和回滚信息。

    机会评分：

    机会价值 =
  市场需求 25%
+ 用户经验优势 20%
+ 未来增长空间 15%
+ 低成本可验证性 15%
+ 长期资产性 10%
+ 现金流可能性 10%
+ 个人兴趣信号 5%

    分数只是排序工具，不代替外部验证或用户决策。

    7. 数据流¶

    OpenClaw 广泛采集
→ 去重、来源和时间标记
→ Hermes 读取信号清单
→ 趋势聚类与反方证据检索
→ 机会卡片
→ 1–2 周低成本实验
→ 外部证据
→ 方向组合更新
→ 继续、调整或停止

    机会卡片必须包含：

    • 机会类型与简述；
    • 支持证据和反对证据；
    • 信息日期和可能失效条件；
    • 与用户经验的组合关系；
    • 最小验证动作、时间和成本；
    • 继续和停止标准；
    • Fact、Inference、Hypothesis 标识。

    8. 方向组合与运行节奏¶

    方向容量上限：

    • 重点观察方向最多 5 个；
    • 验证方向最多 2 个；
    • 主动投入方向最多 1 个；
    • 缺少证据时允许不存在主方向。

    节奏：

    • 每日：3–5 条重要变化，至少 1 条意外发现，区分最新信号和稳定建议。
    • 每周：趋势、3–5 张机会卡片、正反证据、方向热度变化和 1 个最小实验建议。
    • 每两周：完成 1 个真实验。实验可以是岗位分析、访谈、Demo、工具试用、开源互动、内容反馈或服务测试。
    • 每六周：更新方向组合。
    • 每季度：假设清零式复核、信息源效果审查和技术 Stable 基线审查。

    9. Provider 与凭据¶

    9.1 Provider¶

    • 主 Provider：OpenCode Go，用于常态分析、机会卡片、实验规划和复盘。
    • 备用 Provider：DeepSeek 官方 API，用于主 Provider 限额、鉴权、网络或可恢复服务错误。
    • 不在第一次实施中接入第三个 Provider；火山方舟 Coding Plan 保留为后续评估项。

    9.2 凭据约束¶

    • 先完成所有不需要凭据的安装和配置；
    • 第一次需要 OpenCode Go Key 时暂停，通知用户在 PTY 隐藏输入中粘贴；
    • 录入 DeepSeek Key 时重复同一流程；
    • 不从 OpenClaw 复制凭据；
    • 不通过 shell 参数、echo、临时明文文件、日志、Git 或对话传递 Key；
    • 验证输出只报告 Provider 成功或失败，不显示 Key 片段。

    10. 实施形态¶

    10.1 一次性连续实施¶

    用户要求在一次连续执行中完成所有实施阶段。因此实施计划将包含所有安装、配置、私有数据结构、只读桥接、机会发现工作流、新鲜度策略、安全检查和端到端验证。

    阶段仍作为内部安全关卡，但除以下情况外不暂停要求用户逐阶段批准：

    • API Key 隐藏输入；
    • 系统或安全机制要求的升权批准；
    • 设计外的破坏性操作；
    • 需要新的外部账号、付费或对外沟通权限。

    10.2 内部实施关卡¶

    1. 安装与隔离：安装当前稳定版 Hermes，使用 uv 管理的 Python 3.12 与独立 Profile，不修改 OpenClaw。
    2. 非鉴权配置：完成 Provider 结构、私有存储、只读工作区与安全默认值。
    3. 凭据录入：通过用户隐藏输入配置主、备 Provider。
    4. 核心工作流：建立 Signal→Trend→Opportunity→Experiment→Direction 工作流。
    5. 知识桥接：只读接入当前知识库与现有 MCP 查询能力。
    6. 新鲜度与稳定性：建立 known_latest/recommended_stable、Frontier/Stable 和复核策略。
    7. 端到端验证：使用现有采集样本生成机会卡片、意外发现、方向组合和稳定技术建议。
    8. 审计与交付：检查 Git Diff、凭据泄露、OpenClaw 状态、知识库写入和回滚路径。

    11. 故障与降级¶

    • Hermes 安装失败：保留日志，不修改 OpenClaw，优先修复隔离运行时。
    • 主 Provider 失败：在已配置备用 Provider 后尝试降级，不在原因不明时反复请求。
    • 备用 Provider 也失败：保留待处理输入和状态，不丢弃任务。
    • 网络或官方来源不可用：继续使用已验证 Stable 知识，明确标记本次未复核。
    • 官方文档与代码冲突：保留冲突，不自动覆盖稳定知识。
    • Frontier 验证失败：保持 Stable，记录不兼容原因。
    • 只读后端不可用：停止文件工具测试，不降级为无边界本地写入。
    • 现有知识质量检查失败：记录为项目基线，不将既有问题归因于 Hermes 安装。

    12. 安全与隐私¶

    • Hermes 私有数据不放入当前可能公开同步的知识库。
    • 不在私有层之外保存精确金额；如需风险指标，仅保存可支撑月份或风险等级。
    • 远程来源视为不可信输入，不允许网页或文档指令改变系统规则、读取凭据或执行高风险操作。
    • 所有外部发布、联系人、投递、付费、删除和破坏性操作保留人工确认。
    • 保留操作日志和版本历史，但日志不得包含凭据、私人原文或敏感输入。

    13. 验收标准¶

    13.1 安装与隔离¶

    • 当前稳定版 Hermes CLI 能正常运行；
    • 使用受管 Python 3.12 和独立 Profile；
    • OpenClaw 配置、Cron、Gateway、钉钉凭据和运行状态未被修改；
    • raw/ 与知识库生产文件未被 Hermes 修改。

    13.2 Provider 和凭据¶

    • OpenCode Go 能完成最小模型调用；
    • DeepSeek 备用 Provider 配置可解析并可完成验证；
    • 凭据未出现在 shell history、命令参数、项目文件、日志、Git Diff 或验证输出中。

    13.3 机会发现¶

    • 能从现有采集样本生成至少 3 张完整机会卡片；
    • 每张卡片包含来源、日期、正反证据、失效条件、最小实验和停止标准；
    • 输出至少包含 1 条非当前主题的意外发现；
    • 能维护观察、验证和主动方向状态；
    • 项目或 Skill 不被默认设定为系统主轴。

    13.4 新鲜度和稳定性¶

    • 高波动技术结论包含官方来源、适用版本和验证日期；
    • 输出能区分 known_latest 与 recommended_stable；
    • 新版本只进入 Frontier，未验证时不覆盖 Stable；
    • 来源不可用或冲突时能够降级并明确报告不确定性。

    13.5 交付审计¶

    • 执行结束后提供安装位置、Profile、启动方式、工作流和回滚说明；
    • 运行可重复的验证命令；
    • 检查 Git Diff，不修改用户已有的无关变更；
    • 所有未通过项目明确报告，不使用未验证的完成声明。

    14. 非目标¶

    • 不将 SmartInspector 或 Skill 建设设为主轴；
    • 不预设唯一职业方向；
    • 不迁移、删除或接管 OpenClaw；
    • 不建立第二套生产 Cron；
    • 不自动向外部平台发布、投递或发送消息；
    • 不把私人职业、财务或人脉信息同步到 GitHub Pages；
    • 不因当前目标屏蔽热门技术或陌生领域；
    • 不自动追逐 Beta/Canary 版本或未验证社区观点。

    15. 回滚¶

    本方案不修改 OpenClaw。回滚应限定在独立 Hermes Profile、独立运行时和新建的私有工作流。卸载 Hermes、删除 Profile 或删除状态属于破坏性操作，实施计划只能记录回滚命令，不默认执行。
